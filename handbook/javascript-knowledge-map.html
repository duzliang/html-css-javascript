核心执行机制
Execution
执行上下文（Execution Context）★ 高频核心
三种执行上下文
  • 全局执行上下文：程序启动创建，唯一，浏览器中 this=window，Node 中=global/globalThis
  • 函数执行上下文：每次函数调用创建一个新的
  • eval 执行上下文：eval() 内部代码（严格模式下禁用）
每个上下文包含：变量环境（Variable Environment）、词法环境（Lexical Environment）、this 绑定。
调用栈（Call Stack）
LIFO 结构，管理执行上下文。JS 单线程，栈只有一个。

栈溢出：递归无终止条件 → Maximum call stack size exceeded。

尾调用优化（TCO）：ES6 规范要求，严格模式下最后一步 return 函数调用可复用栈帧。V8 未实现，Safari 实现。
词法环境（Lexical Environment）
由两部分组成：
① 环境记录（Environment Record）：存储变量/函数绑定（声明式/对象式/全局/函数/模块）
② 外部引用（outer）：指向外部词法环境 → 构成作用域链

let/const 存于词法环境；var/function 存于变量环境（两者在非函数作用域外部引用相同）。
事件循环（Event Loop）★ 最高频
完整事件循环模型
// 执行顺序框架 // 1. 执行同步代码（Call Stack 清空） // 2. 执行所有微任务（Microtask Queue 清空） // 3. 执行一个宏任务 // 4. 再次清空微任务队列 // 5. 渲染（如需要） // 6. 回到 3 console.log('1'); // 同步 setTimeout(() => console.log('2')); // 宏任务 Promise.resolve().then(() => console.log('3')); // 微任务 console.log('4'); // 同步 // 输出: 1 → 4 → 3 → 2
宏任务 vs 微任务
宏任务（Macrotask / Task）：
  • setTimeout / setInterval / setImmediate(Node)
  • I/O 事件、UI 渲染、postMessage
  • MessageChannel、requestAnimationFrame（rAF）
微任务（Microtask）：
  • Promise.then/catch/finally
  • queueMicrotask()
  • MutationObserver 回调
  • async/await（await 后代码 = .then()）
  • Node：process.nextTick（优先级最高，先于其他微任务）
⚠️关键：每个宏任务结束后，必须清空全部微任务，才进行下一个宏任务。
复杂事件循环题解析 ★ 高频
async/await 与 Promise 混合题
async function foo() { console.log('A'); // 同步 await Promise.resolve(); // 让出控制权（微任务） console.log('B'); // 微任务恢复 } console.log('C'); // 同步 foo(); console.log('D'); // 同步 // 输出: C → A → D → B // await 相当于 .then(cb)，cb进微任务队列
规律：await 之前是同步，await 表达式本身同步求值，await 之后进微任务。每个 await 消耗一个微任务"tick"。
setTimeout(fn, 0) 的真实含义
  • 最小延迟：HTML 规范规定嵌套超过5层后最小4ms；现代浏览器实际约1ms
  • 不是"立即"执行，是"下一个宏任务"执行
  • 多个 setTimeout(fn,0) 按注册顺序执行（同优先级）
  • requestAnimationFrame 在渲染前执行（介于宏任务之间）
  • Node.js 中：process.nextTick > Promise.then > setImmediate > setTimeout
// rAF 与 setTimeout 竞争： requestAnimationFrame(() => console.log('rAF')); setTimeout(() => console.log('timeout'), 0); // 结果不确定（取决于刷新时机）

TDZ（Temporal Dead Zone）：let/const 声明的变量在作用域顶部到声明语句之间，访问会抛 ReferenceError，而非 undefined。这就是"提升了但不能用"的精确描述。typeof 在 TDZ 中的变量也会报错（与 var 不同，var 返回 "undefined"）。

7种原始类型（Primitive）

  • null：有意的空值；typeof null === 'object'（历史 bug）
  • boolean：true / false
  • number：64位 IEEE754 浮点；含 NaN、Infinity、-0

  • bigint：任意精度整数；42n；不能与 number 混合运算

    • 值类型：赋值/传参复制值；不可变（字符串不能修改单个字符）

    类型转换深解 ★ 最高频 难点
    隐式类型转换规则
    ToNumber 规则：
    • undefined → NaN；null → 0；true → 1；false → 0
    • '' → 0；' ' → 0；'3' → 3；'3a' → NaN
    • 对象 → 先 ToPrimitive → 再 ToNumber
    ToString 规则：
    • null → 'null'；undefined → 'undefined'
    • true → 'true'；[] → ''；[1,2] → '1,2'
    • {} → '[object Object]'
    ToPrimitive（对象转原始）：先查 Symbol.toPrimitive，没有则按 hint 调用 valueOf() 或 toString()
    == 宽松相等完整规则
    null == undefined // true（唯一相等对） null == 0 // false！ null == false // false！ NaN == NaN // false（NaN不等于任何值） // 数字 vs 字符串：字符串转数字 1 == '1' // true // 布尔转数字 1 == true // true；false==0 // 对象 vs 原始：对象ToPrimitive [] == 0 // true (''→0) [] == '' // true [] == false // true [1] == 1 // true ('1'→1) {} == '[object Object]' // true
    口诀：null/undefined 只等彼此；对象先转原始；布尔先转数字。
    + 运算符的二义性
    // 有字符串：字符串拼接 1 + '2' // '12' '1' + 2 // '12' true + '1' // 'true1' // 无字符串：数字运算 true + true // 2 true + false // 1 [] + [] // '' (valueOf=>[].toString()=>'') [] + {} // '[object Object]' {} + [] // 0！（{}被解析为空块） +[] // 0 （一元+转数字） +{} // NaN
    falsy / truthy 完整列表
    所有 falsy 值（仅6个+1个）：
    false // 布尔false 0 -0 0n // 零值 '' "" `` // 空字符串 null // 空 undefined // 未定义 NaN // 非数字 // document.all（遗留，特殊）
    注意：'0'、[]、{}、-1、new Boolean(false) 都是 truthy！
    特殊数值深解
    NaN（Not a Number）
    • 类型是 number（typeof NaN === 'number'）
    • 唯一不等于自身的值：NaN !== NaN
    • 任何与 NaN 的运算结果都是 NaN
    • 检测：Number.isNaN(x)（精确）；Object.is(x, NaN)
    • isNaN('abc') = true（先转数字 → NaN，错误用法）
    -0（负零）
    • -0 === 0 → true（===也相等！）
    • -0 > 0 → false；-0 < 0 → false
    • String(-0) → '0'（toString 丢失符号）
    • Object.is(-0, 0) → false（精确区分）
    • 用途：保留运动方向（速度为0时区分方向）
    浮点数精度问题
    0.1 + 0.2 !== 0.3 // true！ // 0.1+0.2 = 0.30000000000000004 // 解决方案： Math.abs(a-b) < Number.EPSILON // 误差比较 (a*10 + b*10) / 10 // 整数运算 a.toFixed(2) // 字符串（注意精度） // Number.MAX_SAFE_INTEGER = 2^53-1 // 超出用 BigInt

    this 绑定规则（完整优先级）★ 最高频 难点
    四种绑定规则（优先级从高到低）
    ① new 绑定（最高）：new Fn() → this = 新建对象
    ② 显式绑定：fn.call/apply/bind(ctx) → this = ctx
    ③ 隐式绑定：obj.fn() → this = obj（调用时点左边的对象）
    ④ 默认绑定（最低）：普通调用 fn() → 严格模式 = undefined，非严格 = globalThis

    ⑤ 箭头函数特殊：无自己的 this，继承定义时外层词法 this，无法被 call/apply/bind 改变。
    const obj = { fn: function() { const arrow = () => this; // 继承fn的this return arrow(); } }; obj.fn(); // obj（箭头函数继承 obj.fn 的 this）
    call / apply / bind 精解
    // call：逐个传参，立即执行 fn.call(ctx, arg1, arg2) // apply：数组传参，立即执行 fn.apply(ctx, [arg1, arg2]) // bind：返回新函数，延迟执行 const bound = fn.bind(ctx, arg1) bound(arg2) // 柯里化式调用 // bind 后 new 的 this 为新对象（bind 失效） function Foo() { console.log(this); } const BoundFoo = Foo.bind({ x: 1 }); new BoundFoo(); // Foo {} 不是 {x:1}
    手写 bind：要处理 new 调用的 this 覆盖（检测 instanceof）。
    函数式编程核心概念
    柯里化（Currying）
    将多参数函数转化为一系列单参数函数：f(a,b,c) → f(a)(b)(c)
    const curry = fn => { const arity = fn.length; return function curried(...args) { if (args.length >= arity) return fn(...args); return (...more) => curried(...args, ...more); }; };
    函数组合（Compose / Pipe）
    // compose: 从右到左执行 const compose = (...fns) => x => fns.reduceRight((v, f) => f(v), x); // pipe: 从左到右执行（更直觉） const pipe = (...fns) => x => fns.reduce((v, f) => f(v), x); // 原生：Array.prototype.flatMap // Promise 链 = pipe
    记忆化（Memoization）
    const memoize = (fn) => { const cache = new Map(); return (...args) => { const key = JSON.stringify(args); if (cache.has(key)) return cache.get(key); const result = fn(...args); cache.set(key, result); return result; }; };
    注意：引用类型参数的 key 需处理（WeakMap 或自定义序列化）。

    Closure & Scope
    闭包完整定义 & 原理 ★ 最高频 核心
    闭包的精确定义
    闭包 = 函数 + 其定义时的词法环境引用。
    当函数在其定义作用域之外被调用时，仍能访问定义时的变量。
    function makeCounter() { let count = 0; // 被闭包捕获 return { inc: () => ++count, get: () => count, reset: () => { count = 0; } }; } const c = makeCounter(); c.inc(); c.inc(); // count=2，但外部不可直接访问
    本质：内部函数持有外部词法环境（EnvironmentRecord）的引用，阻止 GC 回收，变量生命周期被延长。
    经典闭包陷阱 & 解法
    // ❌ var 循环陷阱： for (var i = 0; i < 3; i++) { setTimeout(() => console.log(i), 0); } // 输出: 3 3 3（共享同一个 i） // ✅ 解法1：let（块作用域，每轮新绑定） for (let i = 0; i < 3; i++) { setTimeout(() => console.log(i), 0); } // 输出: 0 1 2 // ✅ 解法2：IIFE 捕获每次的 i for (var i = 0; i < 3; i++) { ((j) => setTimeout(() => console.log(j), 0))(i); }
    作用域链 & 变量查找
    词法作用域 vs 动态作用域
    JS 使用词法作用域（静态作用域）：变量可见性由定义位置决定，非调用位置。

    动态作用域（Bash/早期Perl）：由调用栈决定。

    this 是 JS 中少有的"动态"部分（箭头函数除外），但它不是作用域，是绑定。
    作用域类型
    • 全局作用域：所有函数外；浏览器=window；模块有独立全局
    • 函数作用域：function 内；var 的边界
    • 块作用域：{}（if/for/try等）；let/const/class 的边界
    • 模块作用域：ES Module 每个文件独立作用域
    • eval 作用域：严格模式下有自己的作用域
    闭包内存与泄漏
    • 闭包正常：生命周期合理，变量随闭包释放
    • 内存泄漏场景：
      • 全局变量持有闭包引用（永不释放）
      • DOM 事件未 removeEventListener
      • 定时器未 clearTimeout/Interval
      • 闭包意外持有大对象引用
    • 检测：Chrome DevTools Memory 面板→Heap Snapshot 对比
    • WeakRef + FinalizationRegistry（ES2021）弱引用管理
    IIFE & 模块模式
    IIFE 变体
    // 标准 IIFE (function() { ... })() // 箭头函数 IIFE (() => { ... })() // void IIFE（不关心返回值） void function() { ... }() // 命名 IIFE（调试栈更清晰） (function init() { ... })()
    用途：隔离作用域（避免全局污染）；旧时代模块化替代方案；现已被 ES Module 替代，但在某些场景（如 Top-level await 不可用时）仍有价值。
    模块模式（揭示式）
    const counter = (function() { let _count = 0; // 私有 function _validate(n) { // 私有 return n >= 0; } return { // 公开接口 inc() { _count++; }, get() { return _count; } }; })();
    ES Module 私有实现：未 export 的变量即私有，无需 IIFE。

    • 大对象空间：>1MB 单独分配
    • 代码空间：JIT 编译后的机器码

    面试高频原理题速查 ★ 必背
    问题核心答案
    实现 new 操作符 ① 创建新对象，原型指向构造器.prototype ② 执行构造器，this=新对象 ③ 若构造器返回对象则用该对象，否则返回新对象
    实现 instanceof 沿 obj.__proto__ 链查找，看是否等于 Constructor.prototype；到 null 返回 false
    实现 Object.create 创建空构造函数，其 prototype 设为传入对象，返回 new 该构造函数（或使用 Reflect.setPrototypeOf）
    实现 bind 返回新函数，闭包保存 this 和预传参数；需处理 new 调用时 this 应为新对象而非绑定对象（instanceof 检测）
    实现 deepClone 递归复制；处理循环引用（WeakMap 记录已克隆）；处理 Date/RegExp/Map/Set/Function；推荐 structuredClone
    实现 防抖(debounce) 延迟执行，频繁触发重置计时器；clearTimeout + setTimeout；返回函数需绑定 this 和 arguments
    实现 节流(throttle) 固定时间内只执行一次；时间戳法（立即执行）vs 定时器法（延迟执行）；两者结合保证首尾都触发
    实现 EventEmitter Map 存储事件→监听器数组；on/emit/off/once；once 用包装函数执行后自动 off
    == 与 === 区别 === 无类型转换，类型不同直接 false；== 会触发隐式转换（null==undefined，对象ToPrimitive，布尔转数字）
    typeof vs instanceof typeof：原始类型检测（null='object'的bug），函数='function'；instanceof：原型链检测，跨realm失效，需用 Object.prototype.toString.call
    Promise 链错误为何能被末尾 catch 捕获 每个 .then() 内部有 try/catch 包裹回调；抛出的错误 reject 了该 then 返回的 Promise，沿链传播直到 .catch()
    微任务比宏任务先执行的本质原因 规范要求：每个任务（宏任务）执行完毕后，立即清空微任务检查点（Microtask Checkpoint），才进行下一个任务。微任务是当前任务的"尾声"。
    闭包和垃圾回收的关系 闭包持有外部词法环境引用，阻止 GC 回收该环境中的变量。只要闭包函数本身可达，被闭包的变量就不会被回收。应注意解除不再需要的闭包引用。
