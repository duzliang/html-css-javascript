文档声明与渲染模式 核心

DOCTYPE 声明演进

版本 声明方式 渲染模式 特点
HTML4.01 <!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01//EN" ...> Quirks/Standard 基于SGML，三种DTD
XHTML1.0 <!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN" ...> XML解析 严格的XML语法
HTML5 <!DOCTYPE html> 标准模式 不基于SGML，无DTD
Q: DOCTYPE 的作用是什么？不写的后果？
DOCTYPE 告知浏览器使用何种 HTML 版本进行解析，触发对应的渲染模式。 省略 DOCTYPE 会导致浏览器进入 怪异模式(Quirks Mode)， 以兼容旧版网页的方式渲染，造成盒模型、样式计算等差异。
关键：HTML5 的 DOCTYPE 不区分大小写，且必须位于文档第一行

渲染模式对比

Quirks Mode (怪异模式)

向后兼容旧版浏览器，模拟 IE5.5 的非标准行为。盒模型使用 content+padding+border = width

Standards Mode (标准模式)

W3C 标准行为。盒模型中 width 仅指 content 宽度，符合 CSS 规范

Almost Standards Mode

几乎标准模式，仅表格单元格内图片的垂直对齐有差异

语义化标签 核心 面试热点

HTML5 新增语义标签

<body>
<header> - 页眉/章节头
<nav> - 导航链接区
<main> - 文档主要内容（唯一）
<article> - 独立的内容单元
<section> - 文档章节
<aside> - 侧边栏/附属信息
<footer> - 页脚/章节尾

article vs section vs div

标签 语义 使用场景 是否独立
<article> 独立、完整的内容块 博客文章、新闻、评论卡片 是，可单独分发
<section> 主题性内容分组 章节划分、功能模块 否，依赖上下文
<div> 无语义，纯容器 样式钩子、布局容器 否

其他重要语义标签

<figure> / <figcaption> <time datetime=""> <mark> - 高亮标记 <details> / <summary> <dialog> <address> <cite> <blockquote> / <q> <abbr> <dfn> <kbd> / <samp> / <var>
Q: 为什么需要语义化？
  1. SEO优化：搜索引擎更好理解内容结构，提升排名
  2. 可访问性：屏幕阅读器依赖语义标签提供导航
  3. 代码可读性：开发者能快速理解页面结构
  4. 团队协作：统一的结构规范降低沟通成本
  5. 响应式图片：picture 元素依赖语义结构

Document Outline 算法

HTML5 大纲算法通过 <h1>-<h6> 和 sectioning elements 生成文档目录结构。

  • Sectioning elements: article, aside, nav, section 会创建新的 outline 层级
  • 每个 section 可以有自己的 h1，形成嵌套层级
  • 实际浏览器实现：只有 Safari Reader Mode 等少量工具使用此算法

Meta 标签详解 核心

常用 Meta 标签分类

1. 字符编码与视口

<!-- 字符编码 -->
<meta charset="UTF-8">

<!-- 视口设置（移动端必备） -->
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">
width 视口宽度，device-width 表示设备宽度
initial-scale 初始缩放比例，1.0 表示不缩放
maximum-scale 最大缩放比例（会影响无障碍访问）
user-scalable 是否允许用户缩放（no 会影响无障碍）
viewport-fit 适配刘海屏：contain/cover

2. SEO 相关

<meta name="description" content="页面描述，搜索引擎显示">
<meta name="keywords" content="关键词1, 关键词2">
<meta name="author" content="作者名">
<meta name="robots" content="index, follow">
<link rel="canonical" href="https://example.com/page">

3. Open Graph (社交媒体分享)

<meta property="og:title" content="分享标题">
<meta property="og:description" content="分享描述">
<meta property="og:image" content="https://example.com/image.jpg">
<meta property="og:url" content="https://example.com">
<meta property="og:type" content="website">

4. PWA 相关

<meta name="theme-color" content="#000000">
<meta name="apple-mobile-web-app-capable" content="yes">
<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">
<link rel="manifest" href="/manifest.json">
<link rel="apple-touch-icon" href="/icon-192.png">

5. 安全相关 (CSP)

<!-- 内容安全策略 -->
<meta http-equiv="Content-Security-Policy"
      content="default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'">

<!-- 其他安全头部 -->
<meta http-equiv="X-UA-Compatible" content="IE=edge">
<meta http-equiv="X-Content-Type-Options" content="nosniff">
Q: viewport 中 width=device-width 的原理？
移动设备有两个 viewport：layout viewport（默认约 980px）和 visual viewport（屏幕可视区域）。 device-width 设置的是 layout viewport 的宽度等于设备屏幕宽度， 使页面以 1:1 比例渲染，CSS 像素与屏幕像素对应。

文档结构进阶 核心

Head 中的关键元素顺序

<head>
  <!-- 1. 编码必须在最前（前1024字节内） -->
  <meta charset="UTF-8">

  <!-- 2. IE 兼容模式 -->
  <meta http-equiv="X-UA-Compatible" content="IE=edge">

  <!-- 3. 视口（必须在 CSS 前） -->
  <meta name="viewport" content="width=device-width, initial-scale=1.0">

  <!-- 4. Preconnect / DNS-Prefetch -->
  <link rel="preconnect" href="https://cdn.example.com">
  <link rel="dns-prefetch" href="https://api.example.com">

  <!-- 5. 预加载关键资源 -->
  <link rel="preload" href="/critical.css" as="style">
  <link rel="preload" href="/font.woff2" as="font" type="font/woff2" crossorigin>

  <!-- 6. CSS（避免 @import） -->
  <link rel="stylesheet" href="/styles.css">

  <!-- 7. 其他 meta -->
  <meta name="description" content="...">
  <title>页面标题</title>
</head>

Script 加载优化

属性 加载时机 执行时机 阻塞渲染 使用场景
无 立即 下载完立即 阻塞 关键脚本（已不推荐）
async 并行下载 下载完立即 不阻塞（执行时阻塞） 独立脚本（统计、广告）
defer 并行下载 DOM 解析后 不阻塞 依赖 DOM 的脚本（推荐）
type="module" 默认 defer DOM 解析后 不阻塞 ES Module
nomodule - - - 旧浏览器降级方案
async vs defer 执行顺序：
  • 多个 async 脚本：加载完成顺序 = 执行顺序（不确定）
  • 多个 defer 脚本：按文档出现顺序执行（确定）

表单验证 API 面试热点

约束验证 API (Constraint Validation API)

const input = document.querySelector('input');

// 验证状态属性
input.validity.valueMissing      // 必填但为空
input.validity.typeMismatch      // 类型不匹配（email/url）
input.validity.patternMismatch   // 正则不匹配
input.validity.tooLong / tooShort // 长度超出限制
input.validity.rangeUnderflow / rangeOverflow // 数值超出范围
input.validity.stepMismatch      // 步进值不匹配
input.validity.customError       // 自定义错误
input.validity.valid             // 是否全部有效

// 方法
input.checkValidity()            // 检查有效性，返回 boolean
input.reportValidity()           // 检查并显示错误提示
input.setCustomValidity('错误信息') // 设置自定义错误
Q: 如何实现自定义表单验证样式？
/* 未通过验证时的样式 */
input:invalid {
  border-color: red;
}

/* 已输入内容后的验证（避免初始状态显示错误） */
input:not(:placeholder-shown):invalid {
  border-color: red;
}

/* 通过验证 */
input:valid {
  border-color: green;
}

/* 自定义验证消息样式（有限支持） */
input::placeholder {
  color: #999;
}
注意：:invalid 在页面加载时就会应用，需要配合 :placeholder-shown 或其他技巧

原生表单提交流程

  1. 触发 submit 事件（点击提交按钮或调用 form.submit()）
  2. 浏览器执行 约束验证（检查所有约束属性）
  3. 如果验证通过，触发表单的 submit 事件，然后提交
  4. 如果验证失败，阻止提交，显示第一个无效元素的提示
  5. 调用 event.preventDefault() 可阻止默认提交行为
坑点：调用 form.submit() 方法不会触发表单的 submit 事件，也不会进行约束验证。 如需验证，应使用 form.requestSubmit() 或先调用 form.reportValidity()。

Canvas 高 DPI 适配

function setupHiDPICanvas(canvas, cssWidth, cssHeight) {
  const dpr = window.devicePixelRatio || 1;

  // 设置实际像素尺寸
  canvas.width = cssWidth * dpr;
  canvas.height = cssHeight * dpr;

  // 设置 CSS 尺寸
  canvas.style.width = cssWidth + 'px';
  canvas.style.height = cssHeight + 'px';

  // 缩放上下文
  const ctx = canvas.getContext('2d');
  ctx.scale(dpr, dpr);

  return ctx;
}

localStorage 注意事项

  • 同步阻塞：大量数据操作会阻塞主线程
  • 无过期机制：需要自行实现过期清理
  • 存储事件：同域名下其他页面修改会触发 storage 事件
  • 隐私模式：部分浏览器隐私模式下可能无法使用或数据被清空
  • 异常处理：需捕获 QuotaExceededError
// 安全的 localStorage 封装
const safeStorage = {
  set(key, value, ttl) {
    try {
      const item = {
        value,
        expiry: ttl ? Date.now() + ttl : null
      };
      localStorage.setItem(key, JSON.stringify(item));
    } catch (e) {
      if (e.name === 'QuotaExceededError') {
        // 清理过期项或 LRU 淘汰
        this.clearExpired();
      }
    }
  },
  get(key) {
    const item = localStorage.getItem(key);
    if (!item) return null;
    const parsed = JSON.parse(item);
    if (parsed.expiry && Date.now() > parsed.expiry) {
      localStorage.removeItem(key);
      return null;
    }
    return parsed.value;
  },
  clearExpired() {
    // 实现清理逻辑
  }
};

IndexedDB 现代应用

核心概念

Database（数据库）
Object Store（对象存储）≈ 表
Index（索引）- 加速查询
Record（记录）- { key: value }

基本使用

// 打开数据库
const request = indexedDB.open('myDB', 1);

request.onupgradeneeded = (e) => {
  const db = e.target.result;

  // 创建对象存储
  const store = db.createObjectStore('users', { keyPath: 'id' });

  // 创建索引
  store.createIndex('name', 'name', { unique: false });
  store.createIndex('email', 'email', { unique: true });
};

request.onsuccess = (e) => {
  const db = e.target.result;
  // 进行 CRUD 操作
};

// 使用 Promise 封装（推荐）
async function getUser(id) {
  const db = await openDB('myDB', 1);
  return db.get('users', id);
}

IndexedDB vs localStorage 选择

场景 推荐方案 原因
简单的键值对（token、主题设置） localStorage API 简单，同步访问
大量结构化数据（离线应用数据） IndexedDB 支持索引、事务、大数据量
需要搜索/过滤/排序 IndexedDB 原生索引支持
二进制数据（Blob, ArrayBuffer） IndexedDB 原生支持结构化克隆

<!-- 原生懒加载（现代浏览器） -->

<!-- 配合 Intersection Observer -->
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src;
      observer.unobserve(img);
    }
  });
}, { rootMargin: '50px' });

document.querySelectorAll('img[data-src]').forEach(img => observer.observe(img));

Q: loading="lazy" 和 Intersection Observer 的区别？
  • loading="lazy"：浏览器原生实现，使用简单但控制粒度粗，距离阈值由浏览器决定
  • Intersection Observer：开发者控制，可自定义 rootMargin、threshold，兼容旧浏览器
  • 最佳实践：用 loading="lazy" 作为渐进增强，IO 作为 polyfill 降级方案

Web Components 现代标准

三大核心技术

Custom Elements

自定义 HTML 标签，继承 HTMLElement 类，定义元素生命周期和行为

Shadow DOM

封装的 DOM 子树，与主文档隔离，实现样式和结构的封装

HTML Templates

<template> 和 <slot> 定义可复用的 HTML 模板结构

完整示例

// 定义 Custom Element
class MyButton extends HTMLElement {
  static get observedAttributes() {
    return ['variant', 'disabled'];
  }

  constructor() {
    super();

    // 创建 Shadow Root
    const shadow = this.attachShadow({ mode: 'open' });

    // 使用 template
    shadow.innerHTML = `
      <style>
        :host { display: inline-block; }
        :host([disabled]) { opacity: 0.5; pointer-events: none; }
        button {
          padding: 8px 16px;
          border: none;
          border-radius: 4px;
          background: var(--btn-bg, #0066ff);
          color: white;
          cursor: pointer;
        }
        button:hover { background: #0052cc; }
      </style>
      <button>
        <slot name="icon"></slot>
        <slot>默认文字</slot>
      </button>
    `;

    this._button = shadow.querySelector('button');
  }

  // 生命周期
  connectedCallback() { /* 插入 DOM */ }
  disconnectedCallback() { /* 移除 DOM */ }
  attributeChangedCallback(name, oldVal, newVal) {
    /* 属性变化 */
  }

  // 属性反射
  get variant() { return this.getAttribute('variant'); }
  set variant(val) { this.setAttribute('variant', val); }
}

customElements.define('my-button', MyButton);

特性 open mode closed mode
访问方式 element.shadowRoot 可访问 shadowRoot 返回 null
安全性 可访问，不防开发者 防意外访问，但可绕过
事件 target 重定向到 host 元素 同上
推荐使用 是，便于调试和测试 特殊安全需求
Q: Web Components 与 React/Vue 组件的区别？
维度 Web Components React/Vue
运行环境 浏览器原生 需要框架运行时
渲染方式 命令式 DOM 操作 声明式，虚拟 DOM Diff
数据绑定 手动处理属性/事件 响应式自动更新
生态工具 较少，自行搭建 丰富（路由、状态管理等）
SSR 支持 较弱 完善
总结：Web Components 适合构建跨框架的 UI 组件库（如 Shoelace, Spectrum）， 而框架组件适合构建完整的应用。

PWA 核心技术 现代应用

Service Worker 生命周期

  1. Register：主线程注册，浏览器后台安装
  2. Install：触发 install 事件，适合缓存静态资源
  3. Activate：安装完成，清理旧缓存，开始拦截请求
  4. Idle：等待 fetch/message 事件
  5. Terminate：浏览器回收资源，下次事件重新激活
// sw.js - 缓存策略示例
const CACHE_NAME = 'v1';
const STATIC_ASSETS = ['/index.html', '/app.js', '/styles.css'];

// 安装时缓存静态资源
self.addEventListener('install', (e) => {
  e.waitUntil(
    caches.open(CACHE_NAME)
      .then(cache => cache.addAll(STATIC_ASSETS))
      .then(() => self.skipWaiting())
  );
});

// 激活时清理旧缓存
self.addEventListener('activate', (e) => {
  e.waitUntil(
    caches.keys().then(keys =>
      Promise.all(
        keys.filter(key => key !== CACHE_NAME)
            .map(key => caches.delete(key))
      )
    ).then(() => self.clients.claim())
  );
});

// 拦截请求 - Cache First 策略
self.addEventListener('fetch', (e) => {
  e.respondWith(
    caches.match(e.request).then(response =>
      response || fetch(e.request)
        .then(fetchRes => {
          // 动态缓存
          return caches.open(CACHE_NAME).then(cache => {
            cache.put(e.request, fetchRes.clone());
            return fetchRes;
          });
        })
    )
  );
});

缓存策略

策略 描述 适用场景
Cache First 先读缓存，无缓存再网络请求 静态资源（JS/CSS/图片）
Network First 先网络请求，失败用缓存 实时数据（API 请求）
Stale While Revalidate 立即返回缓存，后台更新 新闻列表、非关键数据
Network Only 只走网络 支付、表单提交
Cache Only 只读缓存 离线页面

HTML 经典面试题 重点

Q: 什么是渐进增强和优雅降级？
  • 渐进增强 (Progressive Enhancement)：从基础功能开始，根据浏览器能力逐步添加增强功能。 先保证核心功能可用，再用新特性提升体验。
  • 优雅降级 (Graceful Degradation)：先构建完整功能，再为旧浏览器提供降级方案。 从高级体验出发，保证基础可用性。

现代开发推荐 渐进增强，配合特性检测（Modernizr）或 @supports 使用。

Q: data-* 属性的作用？与 JSON 存储的选择？

data-* 属性：

  • 存储页面元素相关的轻量级元数据
  • 可通过 element.dataset 访问（驼峰命名转换）
  • 适合存储 ID、状态标记、配置参数等

选择建议：

  • 少量简单数据 → data-* 属性
  • 复杂结构数据 → data-* 存 ID，实际数据放 JS 变量/Store
  • 大量数据 → 完全避免，使用 JavaScript 内存或 IndexedDB
Q: iframe 的优缺点及替代方案？

优点：独立文档、样式隔离、可嵌入第三方内容、实现无刷新上传

缺点：

  • 性能：阻塞主页面的 onload，额外 HTTP 请求
  • SEO：内容难以被搜索引擎索引
  • 体验：无障碍支持差，移动端适配困难
  • 安全：XSS 风险，需配置 sandbox 和 CSP

替代方案：

  • 组件化（React/Vue 组件）替代页面嵌入
  • Shadow DOM 实现样式隔离
  • Micro-frontend 使用 Web Components 或 Module Federation
  • Object/Embed 嵌入特定内容
Q: HTML 解析过程中遇到 script 标签会怎样？
  1. 浏览器解析 HTML，构建 DOM Tree
  2. 遇到 <script> 标签（无 async/defer）：
    • 暂停 HTML 解析
    • 下载脚本（外部脚本）
    • 执行脚本
    • 恢复 HTML 解析
  3. 脚本执行期间，如果操作 DOM，可以访问到已解析的部分
  4. 脚本中访问未解析的元素会得到 null
优化：将脚本放在 </body> 前，或使用 defer/async 属性
Q: 如何理解 HTML 的解析过程（构建 DOM/CSSOM/渲染树）？
  1. 字节流 → 字符流：根据编码（BOM/HTTP Header/meta）解码
  2. Tokenization：将字符流解析为 Token（标签、文本、属性）
  3. 构建 DOM：Token 转换为节点，构建树形结构
    • 遇到开始标签：创建元素，入栈
    • 遇到文本：创建文本节点，挂到当前元素
    • 遇到结束标签：出栈
  4. 并行构建 CSSOM：解析 CSS 文件和 style 标签
  5. 合并 Render Tree：DOM + CSSOM，排除不可见元素（display:none）
  6. Layout（Reflow）：计算元素几何信息
  7. Paint：绘制像素到屏幕
  8. Composite：合成图层（GPU 加速）

SEO 与语义化 重点

SEO 优化要点

标题层级

一个页面一个 h1，标题层级连续不跳级，包含核心关键词

Meta 信息

description 控制在 150 字符内，keywords 已不被 Google 使用

结构化数据

使用 JSON-LD 格式的 Schema.org 标记，支持富媒体摘要

语义化链接

a 标签的 href 要有效，使用有意义的锚文本，避免 "点击这里"

图片优化

alt 属性必须，描述图片内容而非仅 SEO 堆砌关键词

Canonical

重复内容使用 canonical 标签指定规范 URL，避免权重分散

SSR vs CSR 对 SEO 的影响

方案 SEO 友好度 首屏时间 适用场景
SSR（服务端渲染） 优秀 快 内容型站点（新闻、博客、电商）
CSR（客户端渲染） 需配合预渲染 慢（需加载 JS） 强交互应用（后台管理、SaaS）
SSG（静态生成） 优秀 最快（CDN 缓存） 内容不频繁变化的站点
SSR + Hydration 优秀 快 现代主流方案（Next.js, Nuxt）
Q: SPA 应用如何解决 SEO 问题？
  1. SSR/同构渲染：Next.js、Nuxt.js、SvelteKit 等服务端渲染框架
  2. 预渲染 (Prerendering)：构建时生成静态 HTML，如 prerender-spa-plugin
  3. 动态渲染：根据 User-Agent 判断爬虫，提供预渲染版本（如 Rendertron）
  4. Google 的 JavaScript 执行：Googlebot 已能执行 JS，但其他搜索引擎支持有限

无障碍访问 (A11y) 重要

ARIA 属性使用原则

第一原则：能不用就不用

优先使用原生 HTML 语义元素（button、nav、main 等），它们已内置无障碍支持。 ARIA 只应在无法用原生元素实现时使用。

常用 ARIA 属性

role 定义元素角色：button, dialog, tablist, tab, tabpanel 等
aria-label 为元素提供标签（无可见文本时）
aria-labelledby 引用其他元素的 ID 作为标签
aria-describedby 引用补充说明文本
aria-hidden 从可访问树中移除（true/false）
aria-expanded 展开/折叠状态（true/false）
aria-selected 选中状态
aria-live 动态内容更新提示：off/polite/assertive

无障碍检查清单

  • 所有图片有 alt 属性，装饰性图片 alt=""
  • 表单元素有关联的 label
  • 颜色对比度符合 WCAG 标准（4.5:1 正常文本，3:1 大文本）
  • 键盘可完全操作（Tab 导航，Enter/Space 激活）
  • 焦点可见且顺序合理
  • 使用正确的标题层级
  • 动态内容有 aria-live 提示
  • 避免仅通过颜色传达信息

安全相关 核心

常见安全威胁与防护

威胁 原理 防护措施
XSS (跨站脚本) 注入恶意脚本到页面 输入过滤、输出转义、CSP、HttpOnly Cookie
CSRF (跨站请求伪造) 诱导用户发送非预期请求 CSRF Token、SameSite Cookie、验证 Origin
Clickjacking iframe 嵌套欺骗点击 X-Frame-Options、CSP frame-ancestors
MIME Sniffing 浏览器猜测文件类型执行 X-Content-Type-Options: nosniff
HTTPS 降级 中间人攻击强制 HTTP HSTS 头部、预加载列表

CSP (Content Security Policy)

<!-- 通过 Meta 标签 -->
<meta http-equiv="Content-Security-Policy"
      content="default-src 'self';
               script-src 'self' https://trusted.cdn.com 'nonce-xxx';
               style-src 'self' 'unsafe-inline';
               img-src 'self' data: https:;
               connect-src 'self' https://api.example.com;
               font-src 'self' https://fonts.gstatic.com;
               frame-ancestors 'none';
               base-uri 'self';
               form-action 'self';">

<!-- 通过 HTTP Header（推荐） -->
Content-Security-Policy: default-src 'self'; ...
Q: 什么是 SRI (Subresource Integrity)？
SRI 允许浏览器验证引用的外部资源（CDN 文件）是否被篡改。 通过计算文件的哈希值，与 integrity 属性比对。 
<script src="https://cdn.example.com/lib.js"
        integrity="sha384-xxx..."
        crossorigin="anonymous"></script>
如果文件被修改，哈希不匹配，浏览器会拒绝执行，防止 CDN 投毒攻击。

Server-Sent Events (SSE) 实时通信

SSE vs WebSocket

特性 SSE WebSocket
通信方向 服务端 → 客户端（单向） 双向全双工
协议 HTTP（保持连接） WS/WSS（升级协议）
自动重连 原生支持 需自行实现
事件 ID 原生支持 需自行实现
二进制数据 不支持（需 Base64） 原生支持
适用场景 股票行情、日志推送、通知 聊天、游戏、协作编辑

SSE 客户端使用

const eventSource = new EventSource('/sse-endpoint');

// 默认消息（无 event 字段）
eventSource.onmessage = (e) => {
  console.log('Message:', e.data);
};

// 指定事件类型
eventSource.addEventListener('stock-update', (e) => {
  const data = JSON.parse(e.data);
  console.log('Stock:', data);
});

eventSource.onerror = (e) => {
  console.error('SSE Error:', e);
  // 自动重连，可调用 eventSource.close() 停止
};

// 主动关闭
eventSource.close();
